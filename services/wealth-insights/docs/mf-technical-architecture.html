//02Analysis Compute Model

The critical design decision is when does what recompute. Not everything needs to rerun on every NAV refresh. Recomputing everything daily is expensive; recomputing nothing means stale data. The graph must be dependency-aware.

Three compute triggers: (1) CAMS upload — new transaction data, runs full pipeline. (2) NAV daily refresh — re-runs only NAV-dependent modules. (3) Enrichment change — holdings quarterly, fund metadata on change. Each trigger propagates only through affected module subgraph.
// Module dependency graph — arrows show what each module depends on PHASE_1 // pure computation — only reruns on new CAMS data portfolioSummary ← transactions transactionTimeline ← transactions cashflowAnalysis ← transactions sipAnalysis ← transactions taxHarvesting ← transactions, currentNAV // currentNAV reruns daily xirrCalculator ← transactions, currentNAV // currentNAV reruns daily PHASE_2 // enrichment-dependent — reruns when enrichment data changes benchmarkComparison ← xirrCalculator, benchmarkNAV // daily terAnalysis ← portfolioSummary, fundMetadata // on fund metadata change sectorAnalysis ← portfolioSummary, holdingsData // quarterly companyExposure ← portfolioSummary, holdingsData // quarterly assetAllocation ← portfolioSummary, fundMetadata // on fund metadata change marketCapAllocation ← portfolioSummary, holdingsData // quarterly overlapAnalysis ← portfolioSummary, holdingsData // quarterly riskMetrics ← xirrCalculator, benchmarkNAV // daily (rolling window) coverageAnalysis ← portfolioSummary, fundMetadata // on fund metadata change STANDALONE // derived from above — recompute when dependencies change dashboardData ← all Phase1, benchmarkComparison, riskMetrics whatIfScenarios ← xirrCalculator, benchmarkComparison, terAnalysis
Key insight: taxHarvesting, xirrCalculator, benchmarkComparison, and riskMetrics rerun daily because they depend on current NAV. sectorAnalysis, overlapAnalysis, marketCapAllocation rerun quarterly (holdings disclosure cycle). terAnalysis reruns only when AMFI reports a TER change. This means most days, only 5–6 modules actually need to rerun.

ON CAMS UPLOAD (Full Pipeline)

Step 1Parse CAMS → validate, dedupe, merge with existing transactions
Step 2Run all Phase 1 modules synchronously (fast — pure math)
Step 3Kick off enrichment fetch if cache stale
Step 4Run Phase 2 modules async after enrichment resolves
Step 5Run insight evaluator → update insight state documents
Step 6Generate card journeys for newly READY insights
Step 7Run LLM layer (behavioral + narrative) async
TargetPhase 1 done in <2s (user sees loading screen). Full pipeline <30s.

DAILY NAV REFRESH (Partial Recompute)

Step 1Fetch new NAVs for all held schemes via MFAPI.in
Step 2Rerun: xirrCalculator, taxHarvesting (per-lot current value)
Step 3Rerun: benchmarkComparison, riskMetrics (rolling window)
Step 4Re-evaluate trigger conditions (LTCG boundary, NAV milestone)
Step 5Update dashboardData snapshot (hero stats only)
Step 6Rebuild weekly performance card if Monday
TargetAll users processed in background by 7AM IST

//04Insight State Machine
PENDING → READY → SHOWN → DISMISSED or → RE_TRIGGERED
// State transitions PENDING → READY // condition evaluator runs during daily refresh or CAMS upload // checks conditionFn(portfolioAnalysis) → boolean // if true AND (frequencyType=ONCE AND never been SHOWN): set READY // if true AND (frequencyType=TRIGGERED): set READY regardless of history // if true AND (frequencyType=WEEKLY/MONTHLY): set READY on schedule READY → SHOWN // API call: GET /insights → returns all READY insights, ordered by relevanceScore // client marks insight as seen → PATCH /insights/:id/seen → status = SHOWN SHOWN → DISMISSED // user taps "dismiss" or "got it" on card journey Card 5 // for ONCE insights: stays DISMISSED unless retriggerCondition is met SHOWN → SNOOZED // user taps "remind me later" → snoozeUntil = now + 7d (or user-chosen) // condition evaluator skips SNOOZED until snoozeUntil passes DISMISSED → READY // (re-trigger path) // condition evaluator checks retriggerCondition on each run // if met: reset to PENDING → READY on next evaluation // example: behavioral.panicSignals has new entry since shownAt → re-trigger // example: dormant alert — intervalDays=180 has passed → re-trigger

//06Card Journey Assembly

Card journeys are pre-assembled into documents at insight READY time, not at read time. This means the API response is a fetch, not a computation. But card content includes "live bindings" — pointers to portfolio_analysis fields — so numbers stay current after a daily NAV refresh without needing full journey reassembly.

// card_journeys document { insightStateId: ObjectId, userId: ObjectId, assembledAt: Date, cards: [ { position: 1, type: "ANSWER", template: "Your distributor earned {{value}} from your money.", liveBindings: { "value": "phase2.terAnalysis.directVsRegularDelta.totalCommissionPaid" // resolved from portfolio_analysis at read time — stays fresh on daily refresh }, staticData: { fundCount: 6 }, // doesn't change between CAMS uploads verdict: "WARNING", }, { position: 2, type: "EDUCATION", template: "Regular plans have a commission built in...", liveBindings: {}, // education cards are fully static staticData: { ... }, }, // cards 3, 4, 5... ], actions: [ { label: "See which to switch", type: "NAVIGATE", target: "actionable/switch_regular_direct" }, { label: "Ask Artha about this", type: "OPEN_CHAT", context: "regular_plan_cost" } ] }
Live bindings pattern: The card template is static (computed once at READY time), but any number tied to current NAV (current value, current gains, XIRR today) is a binding pointer resolved at API read time from the latest portfolio_analysis. This means the "₹12,400" in card 1 stays accurate on the daily refresh without the whole journey being rebuilt.
//07LLM Integration Layer

Four distinct LLM calls. Each is isolated — they don't chain. They run async after Phase 2 completes. Failures are non-blocking: the product degrades gracefully if an LLM call fails (the rule-based card still shows, LLM narrative is missing).

CALL 1: Behavioral Analyser

InputtransactionTimeline enriched with market events (Nifty index on each transaction date)
OutputinvestorType, panicSignals[], fomoSignals[], missedBuyOpportunities[], cadencePattern
Modelgpt-4o-mini via Vercel AI SDK generateObject() with Zod schema
WhenAfter CAMS upload (Phase 2 complete) + monthly refresh. Never daily.
CacheStored in portfolio_analysis.behavioral. Only rerun if new transactions since last run.

CALL 2: Dynamic Actionable

InputFull portfolio_analysis JSON + list of fixed actionables with their relevance scores
OutputSingle actionable in schema: title, trigger, description, ctaPrimary, ctaSecondary[], dataPoints[]
Modelgpt-4o-mini. Temperature 0.4 for consistency. Schema-validated before storing.
WhenOnly when max fixed actionable relevance score < 60. Never runs if fixed actionables are sufficient.
FallbackSchema validation failure → fall back to next highest fixed actionable. Never surface invalid output.

CALL 3: Weekly Event Explainer

InputFund performance this week + top 3 market events (from BSE/NSE/Ticker news API) + fund holdings
Output2-sentence explanation: why this fund moved this week, in plain Hindi-English
Modelgpt-4o-mini. Simple prompt. Low token cost — runs per user per week.
WhenMonday refresh job, only if fund moved >2% in a week.
FallbackIf no relevant news found: card shows performance number only, no event explanation.

CALL 4: AI Chat (Ask Tab)

InputSystem prompt with full portfolio_analysis JSON + user message + conversation history
OutputStreamed response. May include inline widget triggers (structured tool calls).
Modelgpt-4o or claude-sonnet. Streaming via SSE. Context window: portfolio JSON ~4K tokens.
WhenReal-time on user message. Per-conversation session with portfolio context pre-loaded.
Context loadingIf opened from insight journey: that insight's conditionValue + card content pre-injected as context.
// AI Chat — context pre-loading from insight journey const buildChatContext = (userId, entryPoint: ChatEntryPoint) => { const base = { portfolioSummary: pa.phase1.portfolioSummary, xirrAggregate: pa.phase1.xirrAnalysis.aggregate, taxSummary: pa.phase1.taxHarvesting, behavioralType: pa.behavioral.investorType, topAnomalies: topN(pa.anomalies, 3), }; if (entryPoint.type === "FROM_INSIGHT") { return { ...base, insightContext: { insightKey: entryPoint.insightKey, conditionValue: entryPoint.conditionValue, cardContent: entryPoint.cardJourneyData, // what the user just saw } }; } return base; }; // System prompt injected at session start // "The user just viewed the insight about regular plan costs (6 funds, ₹12,400 in commissions). // They are asking follow-up questions. Answer specifically about their situation."
//08API Design
Principle: The API is thin. It reads from pre-computed state — it does not compute on request. The only exception is the AI chat endpoint, which runs inference in real-time. Every other endpoint is a MongoDB read + live binding resolution.

//09Delivery Layer — Push + WhatsApp

PUSH NOTIFICATIONS

ProviderExpo Push (if React Native) or FCM/APNs direct
TriggerJob runner calls queueDelivery() → writes to a notifications queue (Redis or BullMQ)
WorkerSeparate notification worker processes queue, checks delivery_log for suppression, sends if clear
Payload{ title, body, data: { insightKey, deepLink } } — deep links into the specific card journey
SuppressionBefore sending: check delivery_log for pushSentAt in past 24h. P0 bypasses this check.
LoggingEvery send → delivery_log { userId, channel: "push", insightKey, sentAt, status }

WHATSAPP

ProviderGupshup or Meta Cloud API. Use approved HSM templates for scheduled messages.
Weekly digestComposed by job: portfolio value + 1-line key insight. Plain text, no rich cards. Sent Monday 9AM.
Monthly recapSlightly longer: value, XIRR, fitness score delta, top actionable. Pre-approved template.
Critical triggersELSS unlock in 7 days, LTCG boundary tomorrow — WhatsApp sent same day as push.
Opt-inWhatsApp is opt-in. Consent stored in user preferences. Not sent without explicit consent.
Reply handlingInbound "stop" or "more" handled. "More" → deep link to full insight in app.
// delivery_log — drives all suppression logic { userId: ObjectId, channel: enum("push", "whatsapp", "in_app"), insightKey: string, priority: enum("P0", "P1", "P2"), sentAt: Date, status: enum("sent", "delivered", "opened", "failed"), jobType: string, // which scheduler job triggered this } // Suppression check before any send: async function canSend(userId, channel, priority) { if (priority === "P0") return true; // P0 always sends const recent = await delivery_log.findOne({ userId, channel, sentAt: { $gte: subHours(now(), 24) } }); return !recent; }
//10Infrastructure Stack
🗄️
Database
MongoDB Atlas
Already using it for enriched_cache + folios. TTL indexes for cache expiry. Compound indexes on (userId, insightKey) for state machine. Atlas Search if full-text needed later.
⚡
Job Queue
BullMQ + Redis
Scheduled cron jobs + event-driven trigger jobs in one system. Redis for job state, retry logic, and rate limiting LLM calls. Separate queues per job type for independent scaling.
🚀
API Server
Next.js App Router or Hono
API routes thin — mostly MongoDB reads. Streaming SSE for AI chat. If already on Next.js, App Router handles SSE cleanly. If standalone backend: Hono on Node/Bun is faster.
🤖
LLM
Vercel AI SDK + gpt-4o-mini
Already using generateObject() with Zod for structured output. gpt-4o-mini for behavioral analysis, dynamic actionables, event explainer (low cost, good enough). gpt-4o or claude-sonnet for chat.
📲
Push
Expo + FCM/APNs
If React Native: Expo Push handles token management and platform routing cleanly. If web: FCM directly. Both route through the delivery_log suppression layer.
💬
WhatsApp
Gupshup or Meta Cloud API
Gupshup has pre-approved Indian fintech templates and good reliability for IST-timed messages. Meta Cloud API is cheaper if volume is low (<1000 users). Switch as you scale.
📊
Observability
Axiom or Datadog
Log every job run, insight state transition, delivery event, LLM call (latency + cost). Critical for debugging: "why did user X not get the LTCG boundary alert?" needs full audit trail.
🔐
Auth
Clerk or custom JWT
CAMS data is sensitive. All portfolio_analysis and insight_states are userId-scoped. Every API route validates userId from token before any MongoDB query. No cross-user data access possible.
🌐
Deployment
Railway or Fly.io
BullMQ workers need persistent processes — not serverless. Railway or Fly handle long-running workers cleanly. API can be serverless (Vercel). Separate services for API, workers, and schedulers.
Critical concern — CAMS data privacy: CAMS statement contains PAN, full name, account numbers, complete investment history. Never store raw PDF after parsing. Encrypt portfolio_analysis documents at rest. Restrict MongoDB Atlas network access to known IPs. Implement field-level encryption for PAN and folio numbers. This is non-negotiable before any beta launch.
